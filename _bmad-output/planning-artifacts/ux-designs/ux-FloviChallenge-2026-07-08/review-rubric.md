# Spine Pair Review — FloviChallenge

## Overall verdict

The spine pair is well-shaped and reads at the right depth for a demo-scoped product — tokens are complete, section order is canonical, and the visual/behavioral split is disciplined. But it is not yet a clean source-extraction contract: two of fourteen SPEC capabilities (CAP-9's dispatcher→driver direction, CAP-10 dispatcher-cancel) have **zero** representation anywhere in EXPERIENCE.md — not in IA, not in Components, not in a Flow, not even cited by ID — and CAP-4 and CAP-14 are named in tables but never walked through in a Key Flow. The `sources:` frontmatter paths are also mechanically broken (wrong relative-path depth) and the mockup cross-reference points to a file that doesn't exist at the stated location. These are fixable in under an hour, but a downstream consumer extracting today would build an app with no dispatcher-cancel feature and an unverified realtime-sync direction.

## 1. Flow coverage — thin
Checked: mapped all 14 CAP-N IDs against the 4 Key Flows' numbered steps (not just table mentions), via direct grep of `EXPERIENCE.md` for `CAP-\d+` citations plus manual flow-by-flow reading.

### Findings
- **critical** CAP-10 (dispatcher cancels a request, any status, any time) has no representation anywhere in EXPERIENCE.md — not in Information Architecture, not in Component Patterns, not in State Patterns, not in any Key Flow, and the ID `CAP-10` is never cited (confirmed via grep: 0 hits). The only "Cancel" affordances documented are the edit-modal's dismiss button (`EXPERIENCE.md:68`, "Closes on Save, Cancel, overlay click, or Escape" — cancels the *edit action*, not the request) and the driver-side cancel (CAP-11, `EXPERIENCE.md:72`). A downstream builder has no spec for where/how a dispatcher cancels a request. *Fix:* add a Cancel affordance to the Request card or edit modal in Component Patterns, an IA note, a cancelled-state entry in State Patterns, and at least one flow step demonstrating it.
- **critical** CAP-14 (driver marks a booked gig complete) is named only in the IA table (`EXPERIENCE.md:41`) and the Booked-gig row's Component Pattern (`EXPERIENCE.md:72`, "shows 'Mark complete'... (CAP-14)") — no Key Flow walks through tapping it, and no State Pattern shows the resulting UI change (row removed from Booked gigs, completed-rides count incremented). This matters more than an average missed flow because CAP-14 is what feeds the completed-rides count that CAP-7's and CAP-12's priority mechanic — the demo's most technically distinctive feature — depends on; nothing in the spine shows this mechanic's input action. *Fix:* add a 4th/5th flow beat or extend an existing flow with the mark-complete action and its visible effect.
- **high** CAP-4 (dispatcher edits an existing request) is cited only in the IA table (`EXPERIENCE.md:30`, "or edit (CAP-4) a request") and never demonstrated. Flow 1 only shows *create* (`EXPERIENCE.md:119-125`); no flow clicks "Edit" on an existing card, and no flow shows the pre-filled-modal → save → updated-value-in-list loop that is CAP-4's actual success criterion. *Fix:* extend Flow 1 or add a short flow covering edit specifically, since create and edit are graded as distinct capabilities in SPEC.md.
- **high** CAP-9 (bidirectional realtime propagation) is never cited by ID anywhere (grep: 0 hits) and only half-demonstrated: Flow 3 shows driver→dispatcher propagation (`EXPERIENCE.md:142`, "no page reload"), but no flow shows the dispatcher→driver direction — a newly created/edited request appearing live on a driver's Gigs screen without an app restart, which is also explicitly required by CAP-6's success criterion (`SPEC.md:39`, "a newly created dispatcher request appears without an app restart"). *Fix:* add a step to Flow 2 or a new short flow showing Marcus's Gigs list updating live when a dispatcher creates a request, and cite CAP-9/CAP-6 explicitly.
- **medium** Flow 4 (`EXPERIENCE.md:147-151`) re-demonstrates the identical CAP-7 tie-break mechanic already shown as Flow 2's failure path (`EXPERIENCE.md:134-135`), just with different named characters and no new capability coverage. With only 4 flow slots and CAP-4/CAP-10/CAP-14 uncovered, this is a mis-allocation — the slot spent restating a already-shown mechanic could have covered one of the missing capabilities instead. *Fix:* repurpose Flow 4 to cover CAP-4 (edit) or CAP-14 (mark complete) instead of re-running the same race scenario.
- **low** CAP-11's failure path (cancellation blocked <24h) is captured only in the State Patterns table (`EXPERIENCE.md:86`), not narrated as a "Failure:" line inside Flow 3 the way Flow 1's and Flow 2's failure paths are given inline. Minor since the state is documented, just inconsistent presentation. *Fix:* optional — add a one-line failure branch to Flow 3 for consistency with the other flows' format.

## 2. Token completeness — strong
Checked: extracted all 18 color tokens, 6 typography roles, 5 rounded scale steps, 8 spacing steps, and 5 component tokens from DESIGN.md frontmatter; cross-checked every `{path.to.token}` reference in the prose body and inside the `components` frontmatter object.

### Findings
- **low** `rgba(61,54,48,0.35)` in Elevation & Depth (`DESIGN.md:116`) is a manually-computed value equal to `{colors.text-primary}` (`#3D3630` = rgb(61,54,48)) at 35% opacity, but it's written as a raw literal rather than referencing the token. Not a missing-token problem (it's derivable and correct), but breaks the "reference, don't restate" discipline used everywhere else in the file. *Fix:* note it explicitly as `{colors.text-primary} @ 35% opacity` so the derivation is traceable.
- **low** Typography frontmatter uses non-standard font-weight values (650, 750) against a system-font stack (`DESIGN.md:26-44`) that typically only exposes standard weights (400/500/600/700/800) unless a variable font is loaded. Browsers will silently round these; not fatal, but a downstream dev implementing literal CSS `font-weight: 750` may be surprised it doesn't render as specified. *Fix:* either confirm the intended platform font supports fine-grained weights, or snap to standard weight steps (700/800).

No missing tokens and no missing color hex values were found — this category is otherwise clean.

## 4. State coverage — thin
Checked: walked every IA surface in both Dispatcher web and Driver mobile tables (`EXPERIENCE.md:26-42`) and listed plausible states (cold-load, empty, error, blocked-action, race) against the State Patterns table (`EXPERIENCE.md:78-88`).

### Findings
- **high** No cold-load/loading-skeleton state is defined for any of the three primary list surfaces — Dispatcher Requests, Driver Gigs, Driver Booked gigs. Both calibration examples (Drift, Quill) explicitly model this state (e.g. `experience-example-shadcn.md`, "Cold app load | Today | shadcn Skeleton rows"); Flovi's State Patterns table jumps straight to "empty" states and assumes data is already resolved. *Fix:* add one cold-load row (can be shared across the three list surfaces if the treatment is identical).
- **medium** Notifications (a full dispatcher IA surface, `EXPERIENCE.md:31`) has no "empty" state at all — every other list surface gets one ("no requests yet," "no gigs available," "no booked gigs") but Notifications does not. *Fix:* add "Empty — no notifications yet" to the State Patterns table.
- **medium** Login (both apps) has only a cold/unauthenticated state (`EXPERIENCE.md:80`) — no state for an OAuth failure/denial (network error during Google sign-in, user cancels the Google consent screen). The generic "Network/sync error" row (`EXPERIENCE.md:88`) is scoped to "Either app" post-auth and doesn't clearly cover pre-auth failure. *Fix:* add an explicit auth-failure state or confirm the generic banner is meant to cover it.
- **medium** No validation/save-failure state for the New/Edit Request modal — if a required field is empty or the save network call fails, it's unclear whether the modal stays open with an inline error or closes and shows the generic banner. *Fix:* add a modal-specific error/validation row.
- **medium** No state entry for the result of marking a gig complete (CAP-14) — tied to the same gap noted in Flow coverage. *Fix:* add once the CAP-14 flow gap above is addressed.

## 7. Inheritance discipline — broken
Checked: resolved EXPERIENCE.md's `sources:` frontmatter paths against the actual file locations; grepped for every `CAP-N` citation across the document; cross-checked component names for literal identity across both files.

### Findings
- **critical** Both `sources:` frontmatter paths are mechanically broken. EXPERIENCE.md is at `.../planning-artifacts/ux-designs/ux-FloviChallenge-2026-07-08/EXPERIENCE.md`; the real SPEC.md is at `.../_bmad-output/specs/spec-relocation-dispatch/SPEC.md` — three directory levels up (`../../../specs/...`), but the frontmatter (`EXPERIENCE.md:4-6`) uses only two (`../../specs/...`). Verified computationally: `../../specs/spec-relocation-dispatch/SPEC.md` resolves to a non-existent `planning-artifacts/specs/...` path; the file does not exist there. Same defect affects the `state-machines.md` entry. Any tool that resolves `sources:` literally (rather than a human eyeballing "specs" and guessing) will fail to load either source. *Fix:* change both entries to `../../../specs/spec-relocation-dispatch/SPEC.md` and `../../../specs/spec-relocation-dispatch/state-machines.md`.
- **high** `stack.md` is cited by name in Foundation prose (`EXPERIENCE.md:18`, "per `stack.md`'s hosting decision") — a real, existing file (`_bmad-output/specs/spec-relocation-dispatch/stack.md`) — but is absent from the `sources:` frontmatter list, which names only SPEC.md and state-machines.md. A consumer that trusts the frontmatter as the complete source manifest (the point of the field) would miss a dependency the prose explicitly relies on. *Fix:* add `stack.md` to `sources:`.
- **high** CAP-9 and CAP-10 are never cited anywhere in EXPERIENCE.md (confirmed via exhaustive grep for `CAP-1` through `CAP-14`: every other ID appears at least once, these two appear zero times). This is the traceability symptom of the same gaps flagged in Flow coverage — worth calling out separately here because it means even a targeted "grep for CAP-10" trace-check would falsely conclude the capability was simply omitted from citation, when in fact it has no home in the document at all.
- **medium** Component naming isn't always literally identical across files: DESIGN.md's generic **"Modal"** (`DESIGN.md:128`) corresponds to EXPERIENCE.md's **"New/Edit Request modal"** (`EXPERIENCE.md:68`) — same component, different name, resolvable only by reading both descriptions. Similarly **"Filter chip"** (DESIGN.md:129, singular) vs **"Filter chips"** (EXPERIENCE.md:69, plural). *Fix:* align names exactly, or in Modal's case, keep DESIGN.md generic but have EXPERIENCE.md's row use the same "Modal" name with a "(New/Edit Request)" qualifier.
- **low** IA surface names drift slightly from State Patterns' surface column: "Requests (Overview)" (IA, `EXPERIENCE.md:29`) vs "Dispatcher Requests" (State Patterns, `EXPERIENCE.md:81-82`); "Gigs (browse)" vs "Driver Gigs"; "Booked gigs" vs "Driver Booked". Meaning is unambiguous to a human reader but not literal-string-identical for automated cross-referencing.
