- (decision by user) Paradigm: Postgres-enforced BaaS-direct. No custom backend service; both apps call Supabase directly (supabase-js / supabase_flutter). All shared business rules (role assignment, book/cancel/complete transitions, priority tie-break, auto-reassignment, notification creation) live as Postgres RPC functions + triggers. Clients call RPCs only, never write status/driver_id columns directly. Prevents: two independently-built clients reimplementing the race-sensitive priority rule differently, or bypassing it via direct table writes.
- (decision by user) Role assignment (CAP-1/CAP-5): each app calls its own fixed RPC arg immediately post-OAuth-callback -- dispatcher web always calls claim_role('dispatcher'), driver mobile always calls claim_role('driver'). SECURITY DEFINER RPC creates the caller's profiles row on first call; role column is immutable once set (a later sign-in from the other app cannot flip it). Prevents: the two apps inventing different role-assignment mechanisms, or role being flippable by re-signing-in from the other app.
- (decision by user) CORRECTION: dispatcher visibility/management is OWNER-SCOPED, not a shared pool. A dispatcher can only view, edit, cancel, or otherwise manage relocation requests they created (created_by = self) -- never another dispatcher's requests. This reconciles with CAP-13's own wording ('notified... for one of THEIR requests'), which already implied per-dispatcher ownership; CAP-3's 'views all relocation requests' means all of their own, not a global shared queue. Prevents: RLS/RPCs exposing cross-dispatcher read or write access. Driver-side visibility is UNAFFECTED -- drivers still see the full unbooked pool across all dispatchers (CAP-6 has no per-dispatcher scoping) plus their own bookings.
- (decision) Write boundary detail: column-level privilege revoke on relocation_requests.status and .driver_id from the authenticated client role (only the SECURITY DEFINER RPCs, running as owner, may write those columns); dispatcher client role retains UPDATE on origin/destination/scheduled_date/notes only, gated by RLS created_by = auth.uid(). created_by is set server-side (default auth.uid()), never client-supplied, so a dispatcher cannot spoof ownership on insert.
- (decision) Notifications table scoped by dispatcher_id = owning dispatcher's created_by on the affected request (RLS: recipient = auth.uid()); populated by the reassignment RPC (SECURITY DEFINER insert), not client-derived.
- (decision) Deployment: single Supabase project, no staging/prod split (demo scope). Vue build and Flutter web build (build/web static output) both ship as static sites on Vercel. Single Google OAuth client in Supabase Auth with both apps' URLs allow-listed as redirect URLs. Supabase anon key is safe to bake into both client bundles (RLS-protected); service role key never used client-side.
