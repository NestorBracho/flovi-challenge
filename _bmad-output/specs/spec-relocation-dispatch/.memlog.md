- (decision) Open Q1 resolved (role assignment): role is determined by which app a user signs up through — signup via the dispatcher web app saves role=dispatcher, signup via the driver mobile app saves role=driver. Amends CAP-1 and CAP-5 accordingly.
- (capability) CAP-1 (amended) intent: a user signs in via Google OAuth through the dispatcher web app; signing up through this app persists their account with the dispatcher role. success: unauthenticated visitor redirected to Google OAuth from the dispatcher app; on first sign-in account is saved with role=dispatcher; lands in the dispatcher dashboard.
- (capability) CAP-5 (amended) intent: a user signs in via Google OAuth through the driver mobile app; signing up through this app persists their account with the driver role. success: unauthenticated driver redirected to Google OAuth from the driver app; on first sign-in account is saved with role=driver; lands on the available-gigs screen.
- (assumption) Database seeded with a handful of test dispatcher and driver accounts at initialization for demo convenience; the app-of-signup role rule governs any new sign-ups beyond the seed data.
- (decision) Open Q2 resolved (status lifecycle, delegated to bmad-spec to design): 4-state lifecycle — unbooked (created by dispatcher, visible to drivers), booked (driver claimed or auto-assigned), completed (driver marks done, increments completed-rides count), cancelled (dispatcher-only, terminal). Companion state-machines.md spawned to hold the transition table (companion, not kernel, per diagrams/tables-always-companion rule).
- (decision) Open Q3 resolved (cancellation): dispatcher can cancel a request at any time from any state. Driver can cancel a booked gig only with >=24h minimum notice before the scheduled date; within 24h the attempt is blocked. A valid driver cancellation triggers automatic reassignment to another driver so the dispatcher experience stays smooth; dispatcher is notified in-app when this happens.
- (capability) CAP-10 intent: signed-in dispatcher cancels an existing relocation request at any time, regardless of current status. success: cancelling sets status to cancelled, removes it from driver available/booked views, dispatcher list reflects cancelled status immediately.
- (capability) CAP-11 intent: signed-in driver cancels a gig they've booked, provided at least 24 hours remain before the scheduled date. success: cancellation attempted within 24h of the scheduled date is blocked with a clear message; cancelling with >=24h notice releases the gig for automatic reassignment (CAP-12).
- (capability) CAP-12 intent: when a driver cancels with valid notice, the system automatically re-books the gig to another eligible driver so the dispatcher's view stays uninterrupted. success: within seconds of a valid driver cancellation, the gig is reassigned to the next eligible driver ranked by completed-rides priority (same rule as CAP-7's tie-break), or reverts to the unbooked/available pool if none is immediately eligible.
- (capability) CAP-13 intent: dispatcher is notified in-app when a driver cancellation and automatic reassignment occurs for one of their requests. success: a visible notification/indicator referencing the affected request appears in the dispatcher app when this event occurs, without manual refresh (reuses CAP-9's realtime channel).
- (assumption) Auto-reassignment (CAP-12) draws from all currently active drivers, not a specific waitlist/interested pool, ranked by completed-rides count descending and excluding the cancelling driver — simplification appropriate for demo scope.
- (decision) Open Q4 resolved (concurrent booking): tie-break priority is 'driver with more completed rides wins'. Amends CAP-7 success. This requires a completed-rides count per driver, which only accrues if something marks a ride complete — so a minimal completion capability (CAP-14) is added to make the rule provable, per user's own steer that the lifecycle only needs to prove the concept.
- (capability) CAP-7 (amended) success: tapping book shows confirmation; if two drivers attempt to book the same gig concurrently, the driver with more completed rides is awarded it and the other sees it as no longer available; the winner's gig appears in their booked list.
- (capability) CAP-14 intent: signed-in driver marks a booked gig as completed once the ride has occurred. success: marking complete sets status to completed and increments that driver's completed-rides count (feeds CAP-7 and CAP-12 priority ranking); the gig no longer lists as an active booked gig.
- (assumption) Concurrent-booking priority (CAP-7) is enforced via a database-level check at transaction time comparing driver completed-rides counts, not a distributed queue — true sub-second race conditions are unlikely in a live demo but the rule still governs which write is accepted if two land in the same transaction window.
- (assumption) 'Completed' is a manual driver action (CAP-14 button) rather than automatic/date-triggered — simplest lifecycle that still proves the priority-ranking concept, per user direction that the demo lifecycle need only prove the concept.
